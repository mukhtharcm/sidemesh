---
title: Browser Preview vs Port Forwarding
description: Screenshot stream vs full browser access.
---

Browser preview and port forwarding both let you see web content from the host, but with different fidelity, interaction models, and bandwidth profiles.

## Browser Preview

The daemon launches a headless Chromium instance on the host via CDP, captures screenshots, and streams them to the app. The client can send tap, scroll, text, key, and navigation commands back to the headless browser.

**Best for:**
- Low-bandwidth environments (only compressed screenshots travel the wire)
- Quick visual checks without setting up a tunnel
- Verifying what a headless browser renders on the host
- Situations where the page is heavy (lots of JS, images, fonts) and you only need a visual

**Limitations:**
- Lower fidelity than a real browser (compressed JPEG, periodic frames)
- Interaction is relayed through CDP, not native touch/gestures
- No developer tools, extensions, or native browser features
- Only HTTP/HTTPS targets (no raw TCP)
- The browser runs on the host, so it sees the host's network and user-agent

## Port Forwarding

Port forwarding creates a raw TCP tunnel from the host to the client. The client binds to localhost and uses its native browser.

**Best for:**
- Full native browser experience (mobile Safari, Chrome, etc.)
- Real-time updates from hot-reload (Vite HMR, WebSocket, etc.)
- Testing responsive breakpoints with actual device viewport
- Using developer tools in a real browser
- Tunneling non-HTTP TCP services
- Catching visual regressions (full fidelity, no compression artifacts)

**Limitations:**
- Downloads the full page payload (HTML, CSS, JS, images, fonts)
- Requires a running HTTP server or TCP listener on the host
- Higher bandwidth usage on slow links

## Decision matrix

| Task | Browser Preview | Port Forwarding |
|------|-----------------|-----------------|
| Low-bandwidth / slow links | ✅ (screenshots only) | ❌ (full payload) |
| Heavy pages (lots of JS/assets) | ✅ (screenshots only) | ❌ (downloads everything) |
| Full native browser experience | ❌ | ✅ |
| Native browser dev tools | ❌ | ✅ |
| Real-time hot-reload | ⚠️ (frame updates) | ✅ (native HMR) |
| Non-HTTP TCP services | ❌ | ✅ |
| Tap, scroll, type | ✅ (CDP relayed) | ✅ (native) |
| Check responsive layout | ✅ (dynamic resize) | ✅ (native viewport) |
| Verify host headless browser output | ✅ | ❌ |

## Recommendation

Use **browser preview** when bandwidth is limited, the page is heavy, or you need a quick visual without full browser overhead.

Use **port forwarding** when you need the full native browser experience, real-time hot-reload, developer tools, or when tunneling non-HTTP services.
