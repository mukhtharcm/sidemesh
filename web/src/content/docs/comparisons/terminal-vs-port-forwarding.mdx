---
title: Terminal vs Port Forwarding
description: When to use the integrated terminal and when to use port forwarding.
---

Both the terminal and port forwarding give you access to the host, but they solve different problems.

## Terminal

The integrated terminal spawns a shell on the host and streams stdin/stdout over a WebSocket.

**Best for:**
- Running one-off CLI commands
- Checking system state (`top`, `ps`, `df`)
- Fixing agent mistakes manually
- Interacting with tools that do not expose an HTTP interface
- Git operations the agent cannot perform

**Limitations:**
- No persistent scrollback buffer across reconnects
- Output is plain text; no rich formatting
- Requires the daemon to spawn a shell process

## Port Forwarding

Port forwarding creates a tunnel from a host port to `127.0.0.1` on the client device.

**Best for:**
- Previewing a local dev server on your phone
- Accessing HTTP dashboards (pgAdmin, metrics, etc.)
- Testing responsive layouts in mobile Safari
- Using web-based tools that expect a browser

**Limitations:**
- Only works for TCP/HTTP services
- Cannot run arbitrary shell commands
- Target must be reachable from the host

## Decision matrix

| Task | Terminal | Port Forwarding |
|------|----------|-----------------|
| Run `git rebase` | ✅ | ❌ |
| Preview Vite dev server | ❌ | ✅ |
| Check disk usage | ✅ | ❌ |
| Access pgAdmin | ❌ | ✅ |
| Kill a process | ✅ | ❌ |
| Test on mobile Safari | ❌ | ✅ |
| Edit a config file | ✅ | ❌ |

## Using both together

You can use the terminal to start a dev server and port forwarding to view it. The terminal stays open for logs; the forwarded port handles the HTTP traffic.
